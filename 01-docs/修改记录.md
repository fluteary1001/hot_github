## 2026-05-22：修复历史热点报告链接点击显示"Not Found"问题（二）

### 问题描述

日志显示请求的 `doc_name` 是完整路径 `..\github_projects\01-docs\academic-research-skills-设计说明书.html`，而不是仅文件名。

### 根因分析

前端 `getFileUrl` 函数的文件名提取逻辑有误：

```javascript
// 错误代码
const fileName = path.split('/').pop() || path.split('\\').pop()
```

当路径是 `..\github_projects\01-docs\xxx.html` 时：
1. `path.split('/').pop()` 返回完整路径（因为没有 `/` 分隔符）
2. `||` 短路求值，不会执行 `path.split('\\').pop()`
3. 导致 `fileName` 是完整路径而非仅文件名

### 修改内容

#### frontend/src/pages/HistoryTrendingPage.vue

**修复 `getFileUrl` 函数的文件名提取逻辑**：

```javascript
// 修复后
function getFileUrl(projectId: number, path: string): string {
  if (!path || projectId <= 0) return '#'
  // 先尝试反斜杠分割（Windows路径），再尝试正斜杠分割（Unix路径）
  let fileName = ''
  if (path.includes('\\')) {
    fileName = path.split('\\').pop() || ''
  } else if (path.includes('/')) {
    fileName = path.split('/').pop() || ''
  } else {
    fileName = path // 如果没有分隔符，整个字符串就是文件名
  }
  if (!fileName) return '#'
  const encodedFileName = encodeURIComponent(fileName)
  return `/api/docs/file/${projectId}/${encodedFileName}`
}
```

- 记录 5 个处理步骤的详细信息

## 2026-05-22：修复历史热点报告链接点击显示"Not Found"问题

### 问题描述

历史热点功能中，点击项目的"设计说明书"、"使用说明书"、"价值分析"链接时，返回 `{"detail":"Not Found"}` 错误。

### 根因分析

问题有两层：

1. **前端 URL 编码缺失**：
   - 报告路径包含中文字符（如 `academic-research-skills-设计说明书.html`）
   - 前端 `getFileUrl` 函数直接将文件名拼接到 URL 中，未进行 URL 编码
   - 浏览器发送请求时，中文字符未正确编码，导致后端无法匹配文件名

2. **后端文件名不匹配**：
   - 数据库存储的路径: `..\github_projects\01-docs\academic-research-skills-设计说明书.html`
   - 实际文件名: `academic-research-skills-设计说明书-20260519.html`（带日期后缀）
   - 后端精确匹配失败时无降级方案

### 修改内容

#### 1. frontend/src/pages/HistoryTrendingPage.vue

**修改 `getFileUrl` 函数**：
- 增加 `projectId` 参数，使用实际的 `project.project_id`
- 使用 `encodeURIComponent` 对文件名进行 URL 编码，确保中文字符正确传递

```javascript
// 修复前
function getFileUrl(path: string): string {
  if (!path) return '#'
  const fileName = path.split('/').pop() || path.split('\\').pop()
  return `/api/docs/file/0/${fileName}`
}

// 修复后
function getFileUrl(projectId: number, path: string): string {
  if (!path || projectId <= 0) return '#'
  const fileName = path.split('/').pop() || path.split('\\').pop()
  const encodedFileName = encodeURIComponent(fileName || '')
  return `/api/docs/file/${projectId}/${encodedFileName}`
}
```

**修改模板中的调用**：
- 所有 `getFileUrl(project.reports.xxx)` 改为 `getFileUrl(project.project_id, project.reports.xxx)`

#### 2. backend/app/api/docs_gen.py

**修改 `download_doc` 函数**：
- 增加模糊匹配逻辑：精确匹配失败时，尝试查找前缀匹配的文件
- 按修改时间排序，返回最新的匹配文件

```python
# 新增模糊匹配逻辑
if not doc_path.exists():
    # 尝试查找前缀匹配的文件
    # 例如: "xxx-设计说明书.html" 匹配 "xxx-设计说明书-20260519.html"
    base_name = doc_name.rsplit('.', 1)[0]
    extension = doc_name.rsplit('.', 1)[1] if '.' in doc_name else ''
    
    matching_files = list(docs_dir.glob(f"{base_name}*.{extension}"))
    if matching_files:
        matching_files.sort(key=lambda f: f.stat().st_mtime, reverse=True)
        doc_path = matching_files[0]
    else:
        raise HTTPException(status_code=404, detail="文档不存在")
```

## 2026-05-19：修复历史热点树形显示"第0周"和"0月"问题

### 问题描述

历史热点界面中显示了"第0周"和"0月"，显示错误。按照 trending_history 中数据，实际应该显示 2026-05 和 2026-W21。

### 根因分析

1. **周度数据 month=0**：`_parse_period_value` 函数解析周度数据时，返回 `month=0`，导致周数据无法正确归类到月份节点
2. **查询过滤不严格**：`get_trending_weeks/months/days` 函数使用 `IS NOT NULL` 过滤，但不会过滤 `=0` 的情况，导致月度数据的 `week=0` 和周度数据的 `month=0` 被错误显示

### 修改内容

#### 1. database.py：修复 _parse_period_value 函数

- 周度数据解析时，计算该周周一所在的月份，正确设置 month 字段
- 使用 ISO 周计算逻辑：jan4 定位第1周周一，然后计算指定周的周一日期

```python
# 修复前
elif period_type == 'weekly':
    return year, 0, week, 0

# 修复后
elif period_type == 'weekly':
    jan4 = datetime(year, 1, 4)
    week1_monday = jan4 - timedelta(days=jan4.weekday())
    week_monday = week1_monday + timedelta(weeks=week - 1)
    month = week_monday.month
    return year, month, week, 0
```

#### 2. database.py：修复查询过滤条件

- `get_trending_months`：添加 `AND month > 0` 过滤
- `get_trending_weeks`：从 `week IS NOT NULL` 改为 `week > 0`
- `get_trending_days`：从 `day IS NOT NULL` 改为 `day > 0`

#### 3. 数据修复脚本

修复数据库中已有的错误数据：
```python
# 将 2026-W21 的 month 从 0 改为 5
UPDATE trending_history SET month = 5 WHERE period_value = '2026-W21'
```

## 2026-05-18：修复 Ctrl+C 无法关闭后端的问题

### 问题描述

后端服务在按下 Ctrl+C 时无法正常关闭，日志显示信号处理器被多次调用，进程卡住不退出。

### 根因分析

1. **信号处理器无防护**：每次 Ctrl+C 都会触发信号处理器，导致重复执行关闭逻辑
2. **APScheduler 关闭可能阻塞**：BackgroundScheduler 的 shutdown 方法在某些情况下可能卡住
3. **uvicorn reload 模式**：开发模式下 reload=True 会创建监控进程，信号处理更复杂

### 修改内容

#### 1. main.py：增加关闭防护和详细日志

- 新增 `shutdown_lock` 和 `shutdown_in_progress` 全局变量，防止重复关闭
- 信号处理器中检查是否已在关闭中，避免重复触发
- 增加详细的启动和关闭日志，便于定位问题
- 调度器关闭时用 try-except 包裹，防止异常阻断关闭流程

#### 2. scheduler.py：增强关闭方法

- 关闭前先暂停所有任务
- 增加详细的关闭日志
- 用 try-except 包裹每个操作

#### 3. run.py：增加优雅关闭超时

- 添加 `timeout_graceful_shutdown=5` 参数，5秒后强制关闭

### 涉及文件

| 文件 | 修改类型 |
|---|---|
| `backend/app/main.py` | 增加关闭防护和日志 |
| `backend/app/core/scheduler.py` | 增强关闭方法 |

## 2026-05-17：修复热点页面下载功能 & 增加后端日志

### 问题描述

官方热点(TrendingPage)和高星项目(StarredPage)页面点击"下载"按钮后，只显示"下载任务已启动"提示，实际下载并未真正执行到用户可见的程度——没有进度反馈，也没有完成/失败提示。

### 根因分析

**TrendingPage.handleDownload 和 StarredPage.handleDownload 只调用了 `searchApi.download()`（启动后台任务），但没有轮询任务状态。**

对比 SearchPage 的实现：
- SearchPage：调用 `searchApi.download()` → 获取 `task_id` → 调用 `pollDownloadStatus(task_id)` 轮询到完成/失败
- TrendingPage：调用 `searchApi.download()` → 直接显示"下载任务已启动" → **结束**（不轮询）
- StarredPage：同 TrendingPage

后台任务虽然执行了 git clone，但前端不轮询状态，导致：
1. 用户看不到下载进度
2. 下载成功/失败无反馈
3. `downloading` 状态立即被清除（finally 块），按钮闪烁后恢复

- 新增 `pollDownloadStatus(taskId)` 函数，每秒轮询任务状态直到完成或失败
- 新增 `closeProgress()` 函数关闭进度对话框

## 2026-05-17：AI 文档分析任务持久化追踪

### 问题描述

在项目管理页点击 "AI分析" 后，后台文档生成任务在 FastAPI `BackgroundTasks` 中运行。但关闭进度对话框或切走页签后：
- 前端停止轮询（`stopPolling`），task_id 丢失
- 后台任务仍在运行，但前端无法再追踪进度和结果
- 重新打开页签，无法恢复之前的任务状态

### 根因分析

task_id 和轮询逻辑全部在 `ProjectsPage` 组件的局部变量中，组件销毁即丢失。后端 `doc_tasks` 字典中的状态仍然存在，但前端不再查询。

### 修改内容

#### 1. 新增 stores/docsTask.ts

Pinia store + localStorage 持久化文档生成任务：

- **状态**：`taskId`、`projectId`、`projectName`、`status`、`progress`、`message`、`docs`
- **计算属性**：`isActive` — 判断任务是否仍在进行中
- **start(info)**：启动新任务，写入 store + localStorage
- **resume()**：从 localStorage 恢复 task_id 等信息，返回是否恢复成功
- **clear()**：清理所有状态 + localStorage
- **startPolling()**：每 2 秒轮询 `GET /api/docs/status/{task_id}`，完成/失败时自动停止并清除 localStorage
- **stopPolling()**：停止轮询但保留状态（切页时使用）

#### 2. 改造 ProjectsPage.vue

- 移除组件内的 `analysisStatus` ref、`pollTimer`、`startPolling`、`stopPolling` 局部逻辑
- 改用 `useDocsTaskStore()` 管理所有文档生成状态
- **onMounted**：检查 `docsTask.resume()`，如果有进行中任务则自动弹出进度对话框并恢复轮询
- **onUnmounted**：仅调用 `docsTask.stopPolling()`，不清理 store 状态
- **closeAnalysisDialog**：关闭对话框时，若任务已完成则 `docsTask.clear()`；若仍在进行则只关对话框不清理
- 模板中所有 `analysisStatus` 引用替换为 `docsTask` 对应属性

### 涉及文件

| 文件 | 修改类型 |
|---|---|
| `frontend/src/stores/docsTask.ts` | **新增** 文档任务 Store |
| `frontend/src/pages/ProjectsPage.vue` | 改用 store，支持任务恢复 |

### 效果

| 场景 | 修复前 | 修复后 |
|---|---|---|
| 关闭进度对话框 | 停止轮询，task_id 丢失 | 仅关 UI，后台继续，再次打开仍可追踪 |
| 切走页签再回来 | 停止轮询，丢失追踪 | 自动恢复轮询，弹出进度对话框 |
| 刷新页面 | 完全丢失 | 从 localStorage 恢复 task_id，继续追踪 |
| 任务完成/失败 | 无法知道 | 自动停止轮询，关闭对话框时清理 store |
